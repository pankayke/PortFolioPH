## Core Directives

### 1. Always Verify First
- **Before** making structural changes: Run `flutter test` and `flutter analyze` to establish a baseline.
- **After** any change: Run both commands immediately to ensure no regressions.
- **Baseline Rule:** If a change breaks the GREEN test suite, revert and reassess. Do NOT leave tests red.

### 2. Lint & Deprecation Cleanup
- When addressing analyzer infos, tackle them **systematically** (e.g., all super-parameters first, then deprecations, then dependency warnings).
- Do NOT alter business logic while fixing lints.
- Infos are lower priority than errors/warnings, but should be addressed without risk of regression.

### 3. Test Coverage Rebuilding
When tasked with replacing pruned tests or adding new coverage:
- **STRICTLY inspect** the current domain signatures before writing tests.
- **Use lightweight, focused tests** with mock/fake repositories.
- Do NOT resurrect old data models (e.g., no `JobPostEntity`, `SalaryDuration`, `EmploymentType`, `JobFilters`).
- Pattern: Create test doubles for repositories, stub methods, assert domain rules, verify Either<Failure, Success> contract.

### 4. Zero-Breakage Policy
- **Non-negotiable:** If a change breaks the current GREEN test suite, revert and assess.
- If you encounter conflicts between linting and test integrity, prioritize test integrity.
- Never commit or end a response with red tests.
